## 1 项目概述

### 1.1 编写目的

本系统是基于多模态大模型技术构建的工业设备检修知识检索与作业辅助系统，以提高设备检修知识的检索效率、故障诊断的准确性和作业流程的标准化水平为目标。核心目标是通过大模型的语义理解与生成能力，结合向量检索技术（RAG），实现设备手册、维修案例、操作规程等分散知识的智能化检索与问答，支持文本和图片的多模态输入，为一线检修人员提供精准的知识服务。

为明确系统在业务流程、技术实现、性能指标等方面的具体要求，为后续开发、测试及运维提供可落地的指导框架，特编写本文档。本文档将系统开发中的技术选型、功能规划及场景适配思路进行系统性梳理，为前后端开发、大模型集成、数据库设计提供功能边界与技术标准，确保各模块实现与需求对齐，定义功能验收指标作为开发阶段测试与迭代的基准。

### 1.2 项目背景

#### 1.2.1 传统设备检修知识管理的挑战

工业设备检修是保障生产线稳定运行的关键环节。在传统的检修模式下，知识管理面临以下突出问题：

**（1）知识分散难以检索**

设备手册、维修规程、过往案例等知识散落在纸质文档、PDF文件、个人笔记等不同载体中。一线检修人员在面对故障时，往往需要在大量文档中翻阅查找，耗时费力且容易遗漏关键信息。

**（2）经验传承依赖师徒制**

大量宝贵的检修经验储存在资深技师脑中，通过口传心授的方式传承。当资深人员离职或退休时，这些经验知识也随之流失。新进人员缺乏快速获取经验的有效途径，成长周期长。

**（3）作业流程缺乏标准化**

不同班组、不同人员面对同类故障时，处理流程和操作标准不一致。缺乏统一的作业指引可能导致关键步骤遗漏，影响检修质量和设备安全。

**（4）故障诊断依赖个人能力**

面对复杂故障时，一线人员需凭借个人知识和经验进行诊断。在缺乏有效知识支撑的情况下，容易出现误判和重复操作，延长停机时间。

#### 1.2.2 政策与行业驱动

国家大力推进工业互联网和智能制造战略，《"十四五"智能制造发展规划》明确提出推动人工智能在工业领域的深度应用。设备检修作为工业运维的核心环节，其数字化、智能化转型是实现智能制造的重要基础。在此背景下，将大模型、向量检索等AI技术应用于设备检修知识管理，成为提升工业运维效率的关键路径。

### 1.3 行业痛点与解决方案

#### 1.3.1 行业痛点

**（1）知识检索效率低下**

一线检修人员在故障现场需要快速获取相关知识，但传统的关键词搜索匹配精度差，无法理解用户的自然语言描述，更无法根据故障现象的语义相似度检索历史案例。

**（2）纸质手册数字化程度低**

大量设备手册为PDF格式，其中的图片、表格等非文本内容难以被传统搜索引擎索引。人工提取和标注图片内容工作量大、效率低。

**（3）隐性经验难以沉淀**

检修人员的个人经验（如"某种异响通常是XX故障的前兆"）缺乏便捷的提交和审核机制，难以形成组织级的知识资产。

**（4）AI辅助手段不足**

传统检修辅助工具多为固定问答库，无法理解复杂的故障描述，更不能结合图片进行视觉分析。一线人员急需能理解自然语言、能看图分析、能给出结构化诊断建议的智能助手。

#### 1.3.2 系统解决方案

**（1）智能语义检索**

基于ChromaDB向量数据库和千问Embedding模型，将知识内容向量化后进行语义级别检索。用户以自然语言描述故障现象，系统自动匹配最相关的知识条目，准确率远高于关键词匹配。

**（2）PDF自动解析与图片描述**

使用PyMuPDF（fitz）提取PDF全文和嵌入图片，调用千问VL视觉大模型自动生成图片的中文描述，使图片内容可被检索。

**（3）案例上传与两级审核**

一线人员可便捷提交检修案例（含现场图片），经过管理员初审和专家复审两级审核后纳入知识库，形成经验沉淀闭环。

**（4）RAG智能对话**

采用检索增强生成（RAG）技术，将知识检索结果作为大模型的上下文，生成结构化的故障诊断建议（分析→原因→方案→来源），确保回答基于实际知识而非模型幻觉。

**（5）标准化作业指引**

管理员预设检修流程模板，一线人员按步骤执行任务。每步包含操作说明和合规校验项，确保关键步骤不遗漏。系统自动根据设备型号和检修等级匹配最佳模板。

### 1.4 可行性分析

#### 1.4.1 技术可行性

系统采用成熟的开源技术栈：后端为FastAPI（Python），前端为Vue 3，向量数据库为ChromaDB，大模型API使用DeepSeek和千问。以上技术均具备活跃的社区支持和丰富的文档资源。系统不依赖特定硬件（GPU等），大模型通过云端API调用，降低了本地部署的技术门槛。

#### 1.4.2 经济可行性

系统开发以开源技术为主，无商业软件许可费用。运行成本主要为大模型API调用费用：DeepSeek约¥0.001/千tokens，千问VL约¥0.003/千tokens，Embedding约¥0.0007/千tokens，按中等使用频率估算月成本控制在数百元级别。相比传统人工检索和咨询模式，可显著节省人力成本。

#### 1.4.3 操作可行性

系统采用B/S架构，用户通过浏览器访问，无需安装客户端。界面设计遵循直观简洁原则，用户端导航清晰（首页→AI助手→作业指引→案例上传→客服工单→个人工作台），一线人员可快速上手。管理端功能模块通过侧边栏分类组织，管理员可通过仪表盘快速定位待处理事项。

#### 1.4.4 法律可行性

系统开发与运营严格遵守国家相关法律法规。在数据安全方面，遵循《网络安全法》《数据安全法》《个人信息保护法》的规定，用户密码采用bcrypt哈希存储，API通信使用JWT令牌认证。在知识产权方面，用户上传内容的知识产权归用户所有，系统仅提供平台服务。

## 2 需求分析

### 2.1 目标用户

系统共有三类目标用户：

| 角色 | 标识 | 核心诉求 |
|------|------|---------|
| **一线检修人员** | worker | 快速查找设备检修知识；获得AI故障诊断建议；按标准流程执行检修任务；提交经验案例 |
| **知识管理员** | admin | 管理知识库内容；审核用户提交的案例；管理用户账号；配置系统参数 |
| **技术专家** | expert | 复审经验型案例的技术准确性；回复专业工单；提供权威技术指导 |

### 2.2 业务需求

#### 2.2.1 用户端业务需求

一线人员需要一个集知识检索、AI问答、作业执行、经验提交于一体的综合工作平台：

1. **知识浏览与检索**：按设备型号、故障标签分类浏览；支持自然语言关键词搜索和语义检索
2. **AI智能对话**：支持文本和图片多模态输入；检修模式下给出结构化诊断建议；可对回答质量进行反馈
3. **标准化作业执行**：按模板创建任务；逐步执行并确认合规项；支持暂停/恢复/交接
4. **案例提交与追踪**：提交检修案例（含图片）；追踪审核进度
5. **客服工单**：提交问题工单；查看回复并确认解决

#### 2.2.2 管理端业务需求

管理员和专家需要一个统一的管理后台来维护系统的正常运转：

1. **知识库管理**：新增/编辑/发布/归档/删除知识条目；PDF手册导入解析；版本历史管理
2. **案例审核**：查看案例内容（含图片）；通过/驳回/修改后通过；两级审核（管理员初审+专家复审）
3. **用户反馈管理**：处理客服工单（回复/关闭/转知识条目）；查看AI反馈（有用/无用标注）并转知识
4. **用户管理**：审核通过新用户；启用/禁用账号；重置密码
5. **系统配置**：管理设备型号和故障标签；管理作业模板（含步骤编辑器）；动态配置大模型API

## 3 系统设计

### 3.1 系统架构设计

系统采用前后端分离的B/S三层架构：表现层（Vue 3前端）、业务逻辑层（FastAPI后端）、数据层（SQLite/PostgreSQL + ChromaDB + 文件系统）。

```
┌─────────────────────────────────────────────────────────┐
│                      表现层 (Presentation)               │
```

### 3.5 前端界面设计

#### 3.5.1 界面设计原则

- **直观性**：用户端采用顶部导航+内容区布局，管理端采用侧边栏+内容区布局，功能入口明确
- **一致性**：所有页面统一使用Element Plus组件库，色彩、间距、字体保持一致
- **响应式**：PC Web端优先，适配平板设备
- **反馈清晰**：所有操作有明确的成功/失败提示，空状态有引导文字

## 4 核心技术实现

### 4.1 RAG 检索增强生成链路

系统采用RAG（Retrieval-Augmented Generation）技术实现智能检修问答。其核心流程如下：

```
用户提问 "Y系列电机异常振动可能是什么原因？"
     │
     ▼
┌──────────────┐
│ 1. 意图判断    │ ← 前端 chat_mode 参数 (rag/casual)
└──────┬───────┘
       │ rag（检修模式）
       ▼
┌──────────────┐
│ 2. 向量检索    │ ← ChromaDB 语义检索 Top 5 最相关知识片段
│               │   每条截取600字符，总上下文约3000字符
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 3. 构建Prompt  │ ← 拼接：System Prompt + 知识上下文 + 对话历史(10轮) + 用户消息
│               │   Prompt 模板指定四个输出区域：分析→原因→方案→来源
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 4. 调用大模型  │ ← DeepSeek（文本）/ 千问VL（含图片）
│               │   参数：temperature=0.3, max_tokens=8192, timeout=90s
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 5. 解析回复    │ ← 按标题解析：分析/可能原因/推荐方案/参考来源
│               │   提取为结构化JSON供前端渲染
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 6. 返回结果    │ ← {id, role, content, structured_reply}
└──────────────┘
```

**关键设计决策**：

- **Top K 设为5**：在检索精度和上下文长度间平衡，避免过大上下文挤占生成空间
- **每条600字符截断**：确保总上下文约3000字符，为生成留足~5000 tokens
- **重试机制**：API调用最多重试3次，90秒超时；4xx错误不重试（认证/参数错误）；5xx和网络错误重试
- **模式分离**：闲聊模式跳过向量检索和结构化解析，直接对话式回答

### 4.2 大模型适配层设计

系统通过ModelAdapter抽象基类和ModelRouter路由器实现多模型的统一调用和动态切换。

**设计模式**：

```
                    ┌──────────────┐
                    │ ModelAdapter  │  ← 抽象基类
                    │ (ABC)        │
                    │ chat()       │
                    │ analyze_image│
                    └──────┬───────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
    ┌─────────┴──┐  ┌─────┴──────┐    │
    │DeepSeek    │  │QwenVL     │    │
    │Adapter     │  │Adapter    │    │
    │文本对话    │  │图文理解    │    │
    └────────────┘  └────────────┘    │
                                      │
                        ┌─────────────┴──┐
                        │  ModelRouter   │  ← 路由器：运行时零DB访问
                        │  _db_cfg: dict │
                        │  text/vision/  │
                        │  embedding     │
                        └────────────────┘
```

**配置加载策略**：

1. **启动时一次性加载**：`ModelRouter.load_configs_sync()` 在 `lifespan` 中调用，使用隔离的 `sqlite3.connect()` 读取 `model_configs` 表
2. **运行时零DB访问**：配置缓存在 `_db_cfg` 字典中，避免与SQLAlchemy的写锁冲突
3. **自动类型识别**：根据模型名称关键词（deepseek/chat→text, qwen/vl→vision, embed/bge→embedding）自动归类
4. **配置切换**：管理员修改配置后调用 `invalidate_cache()` 清空缓存重新加载

### 4.3 向量数据库集成

使用ChromaDB作为向量数据库，封装为 `ChromaVectorStore` 服务类。

**Embedding提供者选择优先级**：

1. 数据库配置的Embedding模型（管理员在后台激活）
2. `.env` 中的 `EMBEDDING_PROVIDER=api` → 千问DashScope `text-embedding-v4`（推荐，1024维，中文最优）
3. `.env` 中的 `EMBEDDING_PROVIDER=local` → 本地 `BGE-Small-ZH` 模型（离线可用，需下载~100MB）
4. ChromaDB内置默认 `all-MiniLM-L6-v2`（兜底，中文效果弱）

**关键实现细节**：

- **延迟初始化**：Embedding函数在首次入库或检索时才加载，避免启动阻塞
- **线程安全**：使用 `threading.Lock` 双重检查锁定确保多线程下单次初始化
- **零向量检测**：Embedding调用失败时返回零向量 `[0.0]*1024`，检索时可检测并处理
- **元数据过滤**：支持 `filter_metadata` 参数按设备型号等字段过滤检索结果

### 4.4 PDF 解析与图片描述

使用PyMuPDF（fitz）作为PDF解析引擎，集成千问VL视觉模型生成图片描述。

**解析流程**：

```
PDF文件
   │
   ▼
┌────────────────┐
│ PyMuPDF 解析    │
│ ├ 提取文本      │
│ ├ 提取内嵌图片   │ → 保存为临时PNG
│ └ 分页处理      │
└──────┬─────────┘
       │
       ▼
┌────────────────┐
│ 图片描述生成     │ → 千问VL: "请详细描述这张设备维修手册图片中的内容..."
│ (每张图片)      │
└──────┬─────────┘
       │
       ▼
┌────────────────┐
│ 内容分块向量化   │ → 文本+图片描述 按页分chunk
│ Embedding API  │   存入 ChromaDB
└──────┬─────────┘
       │
       ▼
┌────────────────┐
│ 创建知识条目     │ → KnowledgeEntry(status=draft)
│ KnowledgeEntry  │   等待管理员确认发布
└────────────────┘
```

**设计要点**：

- 图片保存为PNG格式后上传到临时目录，调用千问VL分析
- 千问VL调用失败时图片chunk标记为占位，不阻塞整体导入
- PyMuPDF作为主力引擎，pdfplumber作为备选方案
- 导入的条目初始为草稿状态，需管理员确认后发布

### 4.5 审核流程设计

案例审核采用两级审核机制，根据知识类型走不同流程。

```
用户提交案例
     │
     ├── 事实型(is_experience_based=False): 仅需管理员初审
     │         │
     │         ├── 通过 → 入库(create KnowledgeEntry + Version + 向量化)
     │         │
     │         └── 驳回 → 退回提交者(需填写原因)
     │
     └── 经验型(is_experience_based=True): 需两级审核
               │
               ▼
         管理员初审
               │
               ├── 驳回 → 退回提交者
               │
               └── 通过 → 待复审(专家)
                            │
                            ├── 通过 → 入库
                            │
                            └── 驳回 → 退回提交者
```

**入库流程**（通过后自动执行）：

1. 创建 `KnowledgeEntry`（来源=user_upload，状态=published）
2. 将案例中的图片URL嵌入content末尾（`<img>` 标签）
3. 创建初始版本 `KnowledgeVersion V1.0`
4. 提交数据库事务（释放写锁）
5. 调用 `vector_store.add_documents()` 向量化（在事务外，避免锁冲突）
6. 更新 `case.review_status = APPROVED`，关联 `linked_entry_id`

> **设计要点**：第4步先commit再操作向量库是关键设计——在SQLite环境下，长事务（等待Embedding API响应）会持锁导致其他请求阻塞。

### 4.6 AI 反馈闭环设计

系统设计了完整的"AI回答→用户反馈→管理员审查→知识改进"闭环。

```
用户对AI回复标注"无用" + 填写修正建议
         │
         ▼
  ┌──────────────┐
  │ 反馈数据存储    │ → Message.feedback = "useless"
  │               │   Message.feedback_comment = 用户修正
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ 管理员审查      │ → 管理后台 "AI反馈管理" 页面
  │               │   查看完整提问、回复、修正建议
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ 转化为知识条目  │ → KnowledgeEntry(status=draft)
  │               │   标题=用户提问 + 内容=AI回复+修正建议
  └──────┬───────┘
         │
         ▼
  ┌──────────────┐
  │ 管理员编辑发布  │ → 知识库更新 → 下次RAG检索可查到
  └──────────────┘
```

**连续无用提示**：当用户在同一对话中连续两次标记"无用"时，系统提示建议提交客服工单获取人工帮助。

## 5 软件的法律与安全

### 5.1 法律依据

在软件设计与部署过程中，将相关法律法规纳入考量：

**（1）知识产权与版权法**

- 用户上传的知识内容（案例、手册等）版权归原始权利人所有
- 系统作为平台工具，不对用户上传内容主张版权
- 通过大模型API生成的内容，其版权归属遵循API提供方的服务条款
- 系统源代码归项目组所有

**（2）数据隐私与个人信息保护**

- 依据《中华人民共和国个人信息保护法》，系统仅收集业务必需的用户信息（用户名、姓名、工号、班组）
- 用户密码使用bcrypt单向哈希存储，不可逆
- 用户对话内容存储于数据库中，仅用户本人和管理员可查看
- 上传的图片文件存储在服务器本地，通过URL访问

**（3）网络安全法**

- 依据《中华人民共和国网络安全法》，系统实施用户身份认证（JWT）
- 实施基于角色的访问控制（RBAC），三类角色权限隔离
- API通信可使用HTTPS加密（生产环境Nginx配置SSL证书）

### 5.2 数据安全措施

| 安全层面 | 措施 |
|---------|------|
| **身份认证** | JWT令牌 + bcrypt密码哈希；令牌8小时过期 |
| **访问控制** | RBAC角色权限（worker/admin/expert）；路由守卫+后端依赖校验 |
| **数据加密** | 密码bcrypt哈希（不可逆）；生产环境HTTPS传输 |
| **API安全** | 所有数据操作验证用户归属（如只能访问自己的对话）；跨域CORS白名单 |
| **审计追溯** | 所有关键操作记录到audit_logs表（含操作人、时间、操作类型、详情） |
| **数据备份** | 系统设置中支持手动+自动定期备份 |
| **SQL注入防护** | 使用SQLAlchemy ORM的参数化查询，杜绝拼接SQL |
