# 需求规格说明书（V0.4）

## 1. 引言

### 1.1 项目名称

基于多模态大模型技术的设备检修知识检索与作业系统

### 1.2 项目背景与目标

在钢铁、汽车制造等大型制造业中，设备密集、联动性强，检修知识高度分散于手册、日志与老员工经验中，形成"数据孤岛"，导致故障响应慢、新人上手难、非计划停机损失巨大。本项目以多模态大模型技术为核心，构建一个智能化的设备检修知识检索与作业辅助系统，旨在：

- 打破数据孤岛，实现多模态检修知识的高效检索与复用；
- 提供标准化、步骤化的作业指引，降低操作失误率；
- 支持一线人员贡献经验并持续更新知识库，实现知识数字化沉淀与传承。

### 1.3 适用范围

本系统面向工业、能源、制造领域的设备检修场景，供一线检修人员、班组长、技术专家及知识管理员使用。

### 1.4 系统总体要求

- **架构**：B/S 架构，PC Web 端（响应式设计，适配现场平板）
- **运行环境**：LoongArch 架构服务器 + 银河麒麟高级服务器版 V10（强制性要求）
- **大模型**：支持本地部署或云端 API 接入（如 DeepSeek V4 Pro、千问等），通过配置切换
- **检索架构**：知识图谱 + 向量检索双引擎，保证精确匹配与语义泛化互补

## 2. 用户与使用场景

### 2.1 目标用户

- **用户端**：一线检修人员、班组长 —— 检索知识、使用 AI 助手、执行作业指引、上传经验、咨询客服
- **管理端**：知识管理员、技术专家 —— 管理知识库、审核内容、管理用户、回复客服工单、配置设备模板

### 2.2 核心使用场景

- 一线人员在检修现场遇到设备故障，通过文本描述或拍摄现场照片进行知识检索，获取检修方案和历史案例
- 一线人员按系统提供的步骤化作业指引执行检修，关键步骤确认留痕
- 一线人员将本次检修经验整理为案例上传，经管理员审核后纳入知识库
- 一线人员在 AI 助手无法解决问题时，提交客服工单向管理员或专家求助
- 管理员审核用户上传的经验文档，管理知识库条目（新增、编辑、版本回滚）
- 管理员回复客服工单，将优质问答标注后纳入知识库
- 管理员配置设备型号、检修流程模板、模型参数等

### 3.2 用户端功能说明

#### 3.2.1 登录/注册页面

- 用户输入用户名和密码完成登录
- 新用户填写用户名、密码、姓名、工号、所属班组完成注册
- 注册后进入待审核状态，管理员审核通过后方可登录使用
- 登录后根据角色（用户/管理员）跳转至对应首页
- 支持密码找回（通过管理员重置）

#### 3.2.2 首页（知识库浏览与搜索）

首页采用"浏览+搜索"双模式设计，用户可直接翻阅知识库内容，也可通过搜索精准定位。页面右下方提供AI悬浮对话入口，与AI对话页共享同一套对话引擎。

**左侧分类导航**
- 按设备类型（产线→型号）树形展示，数据来源于知识库条目的设备型号聚合
- 按故障标签分组展示，数据来源于知识库条目的标签聚合
- 点击分类节点过滤中间主区域的知识条目列表
- 分类树支持折叠/展开，默认展示第一级

**顶部搜索栏**
- 支持输入自然语言文本描述故障现象
- 支持拖拽上传图片、点击上传图片（现场故障照片）
- 支持设备型号下拉选择（可搜索）
- 搜索结果在中间区域以列表形式展示，与分类筛选可组合使用

**中间主区域：知识条目卡片列表**
- 以卡片形式展示知识条目，每条包含：标题、摘要、设备型号、故障标签、来源类型、更新时间
- 支持按时间、热度排序
- 支持分页加载
- 无分类/搜索时默认展示全部已发布的知识条目（按更新时间倒序）

**右侧AI悬浮对话**
- 右下角固定AI助手悬浮按钮，点击弹出紧凑对话面板（ChatPanel）
- 面板复用全局对话引擎（useChat composable），与全屏AI对话页共享会话历史和对话逻辑
- 若用户在浏览某条知识时打开面板，自动将该知识条目作为背景上下文注入当前对话
- 面板支持最小化、全屏展开（展开后跳转至AI对话页）
- AI回复中引用的知识条目以超链接形式呈现，点击可跳转至知识详情页

**热门/最近浏览**
- 首页底部区域展示"热门知识条目"（基于检索和浏览频次）
- 展示"最近浏览"（基于localStorage缓存，最多10条）

#### 3.2.3 知识检索结果页

- 以列表形式展示检索结果，每条包含：标题、摘要（高亮关键词）、来源（手册名称+版本号 / 工单编号）、相关度评分
- 结果按相关度降序排列，支持按时间、来源类型二次筛选
- 每条结果支持点击进入详情、下载附件
- 若同时上传了图片和文本，展示图文联合检索结果，标注"视觉匹配"类型的结果
- 无结果时展示建议：尝试更换关键词、提交客服工单、上传案例求助

#### 3.2.4 知识详情页

- 展示知识条目完整内容（富文本 + 图片 + 附件链接）
- 展示元信息：来源、版本号、更新时间、贡献者
- 支持下载附件（PDF 手册片段、图纸等）
- 底部展示"相关推荐"（基于知识图谱关联的其他条目）
- "反馈"按钮：用户可标记"有帮助 / 部分有帮助 / 无帮助"，提交修正建议
- 若当前条目是作业指引类型，提供"开始作业"按钮跳转至作业指引页

#### 3.2.5 AI 对话页（智能问答）与 ChatPanel 组件

系统将AI对话能力封装为可复用的 composable 和组件，支持在多个页面中以不同形态嵌入。

**useChat composable（共享对话引擎）**
- 封装对话核心逻辑：会话创建、消息发送、流式接收、历史管理、反馈提交
- 与后端 `/api/v1/chat` 系列接口交互
- 支持上下文注入（设备型号、当前步骤、关联知识条目等）
- 各页面通过 `useChat(conversationId)` 绑定独立会话，互不干扰

**ChatPanel 组件（紧凑模式，可嵌入任意页面）**
- 提供紧凑的对话面板UI，底部输入框 + 消息列表
- 接收 `context` prop，自动将上下文注入当前对话
- 支持最小化（悬浮按钮）、弹出面板、全屏展开三种状态
- 全屏展开后跳转至 AI 对话页（全屏模式），保持同一会话

**Chat.vue 全屏对话页（完整模式）**
- 左侧对话区：用户输入文本描述问题，支持在对话中粘贴图片
- AI 回复以结构化方式呈现：问题分析 → 可能原因 → 推荐方案 → 参考来源
- 对话区顶部展示当前上下文：当前关联的设备型号、当前正在执行的作业步骤（若有）
- 每条 AI 回复下方提供快捷反馈按钮：有用 / 部分有用 / 无用（带修正输入框）
- 修正内容提交后存入反馈数据集，供后续模型优化
- 若用户连续两次标记"无用"，自动提示"是否需要提交客服工单？"
- 右侧展示当前对话引用的知识来源列表
- 左侧栏展示历史对话列表，支持切换/新建对话
- 与 ChatPanel 使用同一 `useChat` composable，共享底层逻辑

**ChatPanel 嵌入场景**
- 首页：右下角悬浮按钮，浏览知识时可携带当前条目上下文提问
- 作业指引页：步骤详情内嵌入口，携带当前步骤上下文（设备型号、步骤内容、合规要求）提问
- AI 回复中引用的知识条目以可点击超链接呈现

#### 3.2.6 作业指引页（步骤化流程）

**任务创建**
- 用户点击"新建任务"，选择设备型号 + 检修等级（日常检修/定修/大修）
- 系统根据设备型号和检修等级自动匹配推荐流程模板（管理员在后台预配置）
- 用户确认模板后生成任务实例（模板的快照拷贝，防止后续模板修改影响进行中任务）
- 若模板包含预计工时，系统记录截止时间用于超时提醒
- 用户也可选择空白模板手动创建

**任务执行**
- 左侧为步骤导航栏，按阶段分组展示（准备 → 停机 → 拆卸 → 检测 → 修复 → 复装 → 试车）
- 中央为当前步骤的详细说明：操作内容、注意事项、合规校验提醒（如"必须执行能量隔离"、"使用指定扭矩扳手"）
- 合规校验项为红色高亮，需人工勾选确认方可进入下一步
- 每步确认操作自动留痕（确认人、确认时间），不可篡改
- 步骤中内嵌"智能问答"按钮，弹出 ChatPanel 组件，自动携带当前步骤上下文（设备型号、当前步骤内容、合规要求），不中断作业流程
- 顶部展示：任务名称、设备型号、检修等级、当前进度、预计剩余时间

**任务中断与交接**
- 用户可主动暂停任务（填写暂停原因）
- 浏览器关闭时自动保存当前进度，下次打开可继续
- 用户可将进行中任务交接给同班组其他人员（填写交接说明）
- 接收人在"我的任务"中看到被交接的任务，点击"接管"继续执行
- 任务操作日志记录完整交接链

**任务完成**
- 所有步骤确认完成后，任务状态变为"已完成"
- 完成记录（含每步留痕）存入操作日志，不可篡改
- 用户可对本次任务使用的模板进行评价（可选），用于模板优化参考

#### 3.2.7 案例上传页

- 填写案例标题、故障描述（富文本）、检修过程、经验总结
- 支持上传现场图片（多张）：先调用文件上传接口存储图片到服务器，再在案例中引用文件URL
- 支持上传附件（PDF、Word 等）
- 关联设备型号、故障分类标签
- 可选择"是否包含个人经验判断"（标注为经验型知识，审核时需专家复审）
- 提交后进入审核队列，用户可在个人工作台追踪审核状态
- 支持保存草稿

#### 3.2.8 客服工单页（我的提问）

- 展示用户提交的所有客服工单列表：工单号、问题标题、状态（待处理/处理中/已回复/已解决/已关闭）、提交时间
- 支持按状态筛选、按时间排序
- 点击工单进入客服对话详情页

#### 3.2.9 客服对话详情页

- 工单制对话：用户提交问题 → 管理员回复 → 用户补充 → 管理员进一步回复（一问多答形式）
- 每条回复可带附件（截图、手册片段等）
- 用户收到回复后确认"已解决"或"继续追问"
- 已解决的工单，管理员可标注为"纳入知识库"，自动生成知识条目草稿进入审核队列
- 工单关闭后用户仍可查看历史记录

#### 3.2.10 个人工作台

- **我的任务**：展示当前分配或正在执行的作业指引任务，包含任务名称、设备型号、当前进度、开始时间
- **我的上传记录**：展示历史上传的案例列表，包含标题、审核状态（待审核/已通过/已驳回/已通过含修改）、提交时间。驳回的条目展示驳回原因，支持修改后重新提交
- **我的客服工单**：快捷入口，跳转至客服工单页
- **我的修正记录**：展示用户对 AI 回复的标注与修正历史，含原始问题、AI 回复、用户修正内容、提交时间

### 3.3 管理端功能说明

#### 3.3.1 管理端登录页

- 管理员使用专用账号密码登录
- 管理员账号由系统初始化时创建，或由上级管理员在用户管理中创建
- 登录后进入管理后台首页

#### 3.3.2 管理后台首页（仪表盘）

- 展示关键统计卡片：知识库条目总数、今日新增案例、待审核数量、待处理工单数、活跃用户数
- 展示近 7 天知识检索量趋势图
- 展示待处理事项快捷入口：审核队列、客服工单
- 展示最新操作日志概要（最近 10 条）

#### 3.3.3 知识库管理页

- 展示知识条目列表，支持按标题、来源、设备型号、分类、状态（已发布/草稿/已归档）搜索筛选
- 每条展示：标题、来源类型（手册/工单/用户上传）、设备型号、版本号、更新时间
- 支持新增知识条目（手动编写或从审核通过的案例生成）
- 支持编辑已有条目（编辑后版本号自动递增，旧版本保留）
- 支持查看条目版本历史，可对比任意两个版本的差异，支持回滚到历史版本
- 支持将条目归档（不删除，但检索时不再出现）
- 支持批量导入（Excel/CSV 格式的结构化知识数据）

#### 3.3.4 知识编辑/新增页

- 富文本编辑器，支持插入图片、表格、代码块（用于技术参数）
- 填写元信息：标题、设备型号（多选）、故障分类标签、来源（手册名称+版本/工单编号）、检修等级
- 若为作业指引类型，需填写步骤化内容（支持阶段-步骤二级结构），每步可设置合规校验项
- 支持预览发布效果
- 保存为草稿或直接发布

#### 3.3.5 审核队列页

- 展示待审核的上传案例列表：标题、提交人、提交时间、知识类型（经验型/事实型）
- 支持按提交时间、知识类型筛选
- 点击进入审核详情页

#### 3.3.6 审核详情页

- 展示上传案例完整内容（文本、图片、附件）
- 管理员可对内容进行修改标注（建议修改的内容高亮）
- 审核操作为两级：
  - **初审**（管理员）：检查格式完整性、分类正确性、来源可靠性
  - **复审**（专家，仅经验型知识）：验证技术准确性
- 审核结果：通过（直接入库）/ 通过含修改（管理员修改后入库，保留修改记录）/ 驳回（填写驳回原因，退回提交者）
- 通过的知识自动生成知识库条目，向量化后纳入检索索引
- 审核记录留痕（审核人、审核时间、操作类型、修改内容）

#### 3.3.7 客服管理页（工单列表）

- 展示所有用户提交的客服工单：工单号、提交人、问题标题、状态、提交时间、最后回复时间
- 支持按状态筛选：待处理、处理中、已回复、已解决、已关闭
- 支持按提交人搜索
- 管理员可将工单分配给指定专家处理
- 点击进入客服工单详情页

#### 3.3.8 客服工单详情页

- 展示完整的提问-回复对话链，按时间线排列
- 管理员输入回复内容，支持富文本、上传附件
- 回复后可更新工单状态为"已回复"
- 当用户确认已解决后，工单状态自动变为"已解决"
- "纳入知识库"按钮：将工单内容转化为知识条目草稿，自动填写标题和摘要，管理员补充后发布
- 工单超时提醒：待处理工单超过 24 小时未回复，仪表盘高亮提醒

#### 3.3.9 用户管理页

- 展示用户列表：用户名、姓名、工号、所属班组、角色、注册时间、状态（正常/待审核/已禁用）
- 管理员可审核新注册用户（通过/驳回）
- 管理员可禁用/启用已有用户
- 管理员可修改用户角色（一线人员/知识管理员/专家）
- 管理员可重置用户密码
- 支持按班组、角色筛选

#### 3.3.10 设备型号与模板管理页

**设备型号管理**
- 展示设备型号列表（产线、型号名称、分类），支持新增、编辑、删除（有关联知识条目时不可删除，仅可归档）
- 设备型号用于知识库条目关联和检索时的精确过滤，统一术语避免同物异名

**故障标签管理**
- 故障分类以扁平标签形式管理，不再使用树形结构（标签之间的层级关系通过知识图谱表达）
- 展示标签列表（标签名称、关联知识条目数），支持新增、编辑、合并（多个同义标签合并为一个）、删除
- 标签合并后，原关联的知识条目自动迁移到合并后的标签

**检修流程模板管理**
- 展示流程模板列表（模板名称、适用设备类型、检修等级、版本号），支持新增、编辑、删除
- 编辑模板时使用步骤编辑器，按阶段添加步骤（准备→停机→拆卸→检测→修复→复装→试车），每步填写：操作说明、注意事项、合规校验项（可多项）、预计耗时
- 每个合规校验项包含：校验内容描述、是否为强制项（强制项未勾选不可进入下一步）
- 模板支持关联参考知识条目（步骤中可引用相关知识条目ID）
- 模板编辑后自动生成新版本号，旧版本保留但不再被新任务使用
- 已有任务使用模板的快照副本，不受模板更新影响

#### 3.3.11 模型配置页

- 展示当前使用的 LLM 配置：模型名称、API 地址、模型类型（云端/本地）
- 支持切换模型（从预设模型列表中选择）
- 支持配置模型参数：Temperature、最大 Token 数、检索召回数量（Top-K）
- 模型配置变更需管理员二次确认
- 支持查看模型调用统计：近 7 天调用次数、平均响应时间

#### 3.3.12 操作日志页

- 展示系统操作日志列表：操作时间、操作人、操作类型、操作对象、IP 地址、操作详情
- 支持按时间范围、操作人、操作类型筛选
- 关键操作类型包括：用户审核、知识条目修改、权限变更、模型配置变更、审核操作、数据导出
- 日志为只读，不可删除或修改
- 支持导出日志为 CSV

#### 3.3.13 系统设置页

- 数据备份：手动触发数据库备份，查看备份历史，配置自动备份周期
- 系统信息：展示版本号、运行环境信息、依赖服务状态（数据库、向量库、模型服务）
- 文件存储配置：存储路径、最大上传文件大小

## 4. 业务规则与逻辑

### 4.1 用户注册与审核流程

1. 用户在登录页点击注册，填写用户名、密码、姓名、工号、所属班组
2. 注册成功后账号状态为"待审核"，用户不可登录
3. 管理员在用户管理页审核新用户：通过则用户可正常登录；驳回则填写原因
4. 管理员可随时禁用已有用户，禁用后用户无法登录，已有数据保留

### 4.2 知识上传与审核流程

1. 一线人员在案例上传页提交案例，案例状态为"待初审"
2. 管理员在审核队列中看到待初审案例
3. 管理员进行初审（格式完整性、分类正确性）：
   - 若案例标注为"经验型"，初审通过后进入"待复审"状态，由专家进行技术准确性审核
   - 若案例为"事实型"，初审通过后直接入库
4. 驳回的案例，提交者可在"我的上传记录"中查看驳回原因，修改后重新提交
5. 通过的案例自动：生成知识条目 → 向量化 → 纳入检索索引 → 更新知识图谱关联

### 4.3 知识版本管理规则

- 管理员编辑已有知识条目时，系统自动生成新版本号（主版本.次版本，如 V1.0 → V1.1 为小改，V1.0 → V2.0 为涉及流程/结论变更的大改）
- 旧版本保留在版本历史中，不被检索索引引用
- 管理员可查看任意两个版本的差异对比
- 管理员可将当前版本回滚至任一历史版本（回滚操作记录在操作日志中）

### 4.4 客服工单流程

1. 用户在 AI 对话中连续两次反馈"无用"时，系统提示是否提交客服工单；用户也可从客服工单页直接提交
2. 用户填写问题标题、详细描述，可上传辅助图片，提交工单
3. 工单状态为"待处理"，管理员在客服管理页看到
4. 管理员接手工单（状态变为"处理中"），回复用户
5. 用户收到回复后可"确认解决"（状态变为"已解决"）或"继续追问"（状态变为"处理中"）
6. 已解决的工单，管理员可点击"纳入知识库"生成知识条目草稿
7. 管理员可关闭工单（状态变为"已关闭"），已解决的工单 7 天后自动关闭

### 4.5 AI 对话反馈规则

- 每条 AI 回复用户可反馈：有用 / 部分有用 / 无用
- 选择"无用"或"部分有用"时弹出修正输入框，用户可填写修正内容
- 反馈数据存入反馈数据集，用于后续模型微调或 Prompt 优化时的参考

### 4.6 作业指引确认留痕规则

- 合规校验项为强制性勾选，未全部确认不可进入下一步
- 每步确认记录包含：确认人、确认时间、步骤编号、步骤名称
- 确认记录与任务绑定，任务完成后存入操作日志，不可篡改
- 任务中途退出时，已完成的确认记录保留，恢复任务时可继续

### 4.7 设备型号与知识关联规则

- 知识条目可关联一个或多个设备型号
- 删除设备型号时，若有关联的知识条目，提示不可删除，仅可归档
- 检索时设备型号筛选为精确匹配，文本搜索为语义匹配，二者取交集

### 4.8 初始知识库构建（PDF/文档离线解析管线）

本系统的初始知识库主要来源于设备检修手册（PDF 格式），这些 PDF 通常包含文字、结构图、剖面图、数据表格等混合内容。需要在系统部署时进行一次性离线解析入库。

**技术方案**：
- **PDF解析库**：PyMuPDF (fitz) — 提取文本、表格、内嵌图片；纯Python实现，LoongArch兼容性较好。若PyMuPDF不可用，备选 pdfplumber（纯Python，图片提取能力弱）或 pypdf
- **图片描述生成**：调用千问VL API，输入PDF中导出的图片，生成中文描述文本
- **文本分块**：按章节+页面分块，保留来源页码和版本信息
- **向量化入库**：分块文本 → Embedding模型向量化 → ChromaDB向量库

**处理流程**：

1. **文档预处理**：PDF 拆分为页面，检测每页的图文区域（文字块、图片块、表格块）
2. **多模态提取**：
   - 文字块 → 直接提取文本，保留章节结构（标题层级）
   - 表格块 → 提取为结构化 JSON（后续可供结构化查询）
   - 图片块（剖面图、结构图、热像图等）→ 调用视觉大模型（千问 VL）生成图片描述文本，描述内容包括"图中展示了什么组件、标注了什么信息、反映了什么故障特征"
3. **图文对生成**：将同一章节/页面的文字与图片描述合并，形成完整的知识片段（Chunk），保留来源页码和版本信息
4. **知识图谱实体抽取**：对每个知识片段，调用大模型抽取实体（设备名称、组件、故障现象、症状、原因、检修方案）及关系，生成初始知识图谱三元组
5. **向量化入库**：知识片段文本 → Embedding 模型向量化 → 存入 ChromaDB 向量库；图片原文及描述存入文件存储，数据库记录映射关系
6. **人工校验环节**：抽取结果（特别是知识图谱三元组）需经管理员在审核界面中校验确认后方可发布。系统提供批量审批和逐条审批两种模式

该管线在后续增量更新时可复用：当管理员上传新版手册时，触发增量解析流程。

### 4.9 知识图谱初始化与维护策略

知识图谱是保证精确匹配的基础，初始化策略如下：

1. **初始构建**：从 PDF 手册中半自动提取（见 4.8），人工校验后入库
2. **增量更新**：审核通过的案例自动抽取实体及关系，作为新节点/边加入图谱
3. **实体类型定义**：
   - **设备**（如"XX 型柴油发动机"）
   - **组件**（如"活塞"、"曲轴"、"涡轮增压器"）
   - **故障现象**（如"冒黑烟"、"异响"、"过热"）
   - **故障原因**（如"喷油嘴堵塞"、"轴承磨损"）
   - **检修方案**（如"更换喷油嘴"、"调整气门间隙"）
   - **合规要求**（如"必须执行能量隔离"）
4. **关系类型定义**：
   - `设备 -[包含]-> 组件`
   - `组件 -[可能发生]-> 故障现象`
   - `故障现象 -[可能由...导致]-> 故障原因`
   - `故障原因 -[解决方案]-> 检修方案`
   - `检修方案 -[需遵守]-> 合规要求`
5. **图谱检索应用**：知识检索时，优先在图谱中精确匹配实体，再结合向量检索做语义扩展。例如用户输入"发动机冒烟"，系统先在图谱中定位"发动机"→"冒黑烟"→可能原因列表，再通过向量检索查找相关方案详情
6. **图谱质量维护**：管理员定期审查孤立节点（无关联的实体）、冲突关系（同一现象指向矛盾原因），人工修正

### 4.10 作业指引异常中断处理

现场作业可能因各种原因中断，系统需保证作业状态可恢复、可交接：

1. **中断场景识别**：
   - 用户主动暂停（点击"暂停任务"）
   - 浏览器关闭/标签页关闭（前端检测 beforeunload 事件，记录最后确认步骤）
   - 长时间无操作（超过 30 分钟无步骤操作，会话过期前自动保存进度）
2. **中断记录**：中断时记录当前步骤编号、已完成的合规确认项、中断时间、中断原因（若用户主动填写）
3. **恢复机制**：
   - 用户在"我的任务"中点击"继续任务"，从上次确认的最后一步继续
   - 已完成的合规确认项保留且不可篡改，无需重新确认
   - 中断超过 24 小时的任务，恢复时再次弹出"当前设备型号、检修等级"确认，防止记忆偏差
4. **交接机制**：
   - 用户可将进行中的任务"交接"给同班组其他人员
   - 交接时附交接说明（富文本，可简述当前进展和注意事项）
   - 接收人在"我的任务"中看到被交接的任务，点击"接管"
   - 任务操作日志记录交接链（原执行人→接收人、交接时间）
5. **超时提醒**：任务开始后超过预计工时（由模板配置）未完成，系统向执行人发送提醒（仪表盘通知）；超过预计工时 2 倍仍未完成，同时通知管理员

## 5. 大模型助手运作架构

本章描述 AI 助手从接收用户输入到生成回复的完整技术链路，包括架构选型、工具调用、模型路由以及与前后端各组件的交互方式。

### 5.1 总体架构：RAG + Agent

系统采用 **RAG（Retrieval-Augmented Generation，检索增强生成）** 作为核心架构，并在其上叠加 Agent 模式处理复杂多步任务。整体链路分为五个阶段：

1. **输入预处理**：前端收集用户输入（文本 + 图片 + 设备型号参数），判断输入类型并路由
2. **意图识别与重写**：大模型分析用户意图，将口语化、碎片化的现场描述重写为结构化查询
3. **多路检索**：同时触发知识图谱精确查询和向量语义检索，检索结果融合排序
4. **上下文增强**：将融合后的检索结果作为 Context 注入 Prompt，附带当前上下文（设备型号、作业步骤等）
5. **生成与校验**：大模型生成结构化回复，后处理校验（来源标注、合规检查），返回前端渲染

### 5.2 模型路由策略

系统不绑定单一模型，通过 ModelAdapter 抽象层按请求类型智能路由：

| 请求类型 | 使用模型 | 路由条件 |
| -------- | -------- | -------- |
| 纯文本知识检索（语义理解+重写） | DeepSeek | 用户仅输入文字 |
| AI 对话（纯文本） | DeepSeek | 对话中未上传图片 |
| AI 对话（含图片） | 千问 VL | 对话中上传了现场图片 |
| 故障图片视觉分析 | 千问 VL | 检索请求中包含图片 |
| 知识图谱实体抽取 | DeepSeek | 批量离线处理 |
| PDF 图片描述生成 | 千问 VL | 离线文档解析管线 |
| Prompt 中的 Embedding 向量化 | Embedding 模型（BGE/M3E） | 知识入库和检索始终使用 |

路由规则由管理端模型配置页控制，管理员可随时调整。路由决策在后端透明执行，前端无感知。

### 5.3 检索增强生成（RAG）详细流程

#### 5.3.1 检索请求处理（/api/v1/search）

1. 前端发送 `{ query: string, image?: File, device_model?: string }`
2. 若有图片 → 千问 VL 提取图片特征描述文本（如"气缸壁有明显划痕，深约 1mm，位于上止点附近"）
3. 意图重写（DeepSeek）：将"机器冒烟了还有点响"重写为"发动机冒烟伴随异响故障排查"
4. **多路检索并行**：
   - 知识图谱查询：定位实体→获取关联故障原因和方案列表
   - 向量检索：查询 ChromaDB 返回语义最相关的 Top-K 知识片段
5. **融合排序**：
   - 知识图谱精确匹配结果 → 权重 1.0
   - 向量语义匹配结果 → 权重 0.7 × 相似度分数
   - 若指定了设备型号 → 精确过滤，仅保留匹配设备的知识
6. 返回排序后的结果列表，每条标注来源和匹配类型（精确匹配/语义匹配/视觉匹配）

#### 5.3.2 对话请求处理（/api/v1/chat）

1. 前端发送 `{ conversation_id: string, message: string, image?: File, context?: { device_model?: string, task_step?: string } }`
2. 若有图片 → 千问 VL 视觉分析生成描述
3. 获取对话历史（最近 N 轮，保留上下文连续性）
4. **检索相关背景知识**：用当前消息 rewrite 后检索知识库，获取 Top-K 知识片段（不直接展示给用户，作为生成素材）
5. **Prompt 拼装**（核心环节）：

    ```text
    System: 你是一个设备检修专家助手。回答时按以下结构组织：
      1. 问题分析（对用户描述的现象进行分析）
      2. 可能原因（列出 1-3 个最可能的原因，按概率排序）
      3. 推荐方案（针对每个原因给出具体操作步骤）
      4. 参考来源（列出引用的知识来源）
    
    当前上下文（如有）：设备型号={device_model}，当前作业步骤={task_step}
    遵守合规要求：所有操作建议必须包含安全提醒。
    
    已知相关知识（从知识库检索获得）：
    {retrieved_knowledge_context}
    
    对话历史：
    {conversation_history}
    
    用户消息：{user_message}
    ```

6. 大模型生成回复（千问 VL 含图或 DeepSeek 纯文本）
7. 后处理：拆解结构化回复（分析/原因/方案/来源），来源编号与检索结果对应，前端可点击跳转
8. 返回 `{ reply: { analysis, causes[], solutions[], sources[] }, conversation_id }`

### 5.4 Agent 模式（复杂多步任务）

对于需要多步工具调用的复杂查询（如"对比型号 A 和型号 B 的曲轴检修流程差异"），系统使用 LangChain Agent 模式：

**Agent 可调用的工具（Tools）**：

| 工具名称 | 功能 | 调用方式 |
| -------- | ---- | -------- |
| `search_knowledge` | 在知识库中检索指定关键词/问题 | 内部调用向量检索 API |
| `query_knowledge_graph` | 在知识图谱中查询指定实体的关联信息 | Cypher 或 API 查询知识图谱 |
| `get_device_spec` | 获取指定设备型号的技术参数 | 数据库查询 |
| `get_task_template` | 获取指定设备+检修等级的作业流程模板 | 数据库查询 |
| `search_similar_case` | 按故障现象查找相似历史案例 | 向量相似度搜索 |
| `compare_procedures` | 对比两个检修方案的差异 | 调用大模型进行文本对比 |

Agent 执行流程：

1. 用户输入复杂问题
2. Agent 规划需要的工具调用序列
3. 依次执行工具调用，获取中间结果
4. 综合所有中间结果，调用大模型生成最终回答

Agent 模式由后端根据用户问题的复杂度自动判断——若单一检索即可覆盖的问题走 RAG 链路，若需要多步推理和多方信息整合则走 Agent 链路。判断逻辑：先做意图分析，若意图涉及"对比"、"为什么...而不是"、跨多个设备或多个故障类型的查询，则触发 Agent 模式。

### 5.5 前端-后端交互协议

| 接口 | 方法 | 路径 | 说明 |
| ---- | ---- | ---- | ---- |
| 知识检索 | POST | `/api/v1/search` | 文本+图片+设备参数，返回排序结果列表 |
| AI 对话 | POST | `/api/v1/chat` | 发送对话消息，返回结构化回复 |
| 新建对话 | POST | `/api/v1/conversations` | 创建新的对话会话 |
| 对话列表 | GET | `/api/v1/conversations` | 获取用户的历史对话列表 |
| 对话详情 | GET | `/api/v1/conversations/{id}` | 获取指定对话的完整历史 |
| 对话反馈 | POST | `/api/v1/conversations/{id}/feedback` | 提交对某条回复的反馈和修正 |
| 图片分析 | POST | `/api/v1/images/analyze` | 单独对图片进行故障特征分析（千问 VL） |
| 文件上传 | POST | `/api/v1/files/upload` | 上传图片或附件，返回文件 URL |

### 5.6 前端对话架构：useChat + ChatPanel

对话AI能力通过 Vue composable 模式实现"一处编写，多处复用"：

**useChat composable（共享对话引擎）**
- 封装对话核心状态与方法：`messages`（消息列表）、`send()`（发送消息）、`submitFeedback()`（提交反馈）、`startNewConversation()`（新建会话）
- 调用后端 `/api/v1/conversations` 系列接口，管理会话生命周期
- 接受 `context` 参数注入背景信息（设备型号、当前步骤、关联知识条目等），上下文自动打包为 Prompt 前缀
- 每个组件实例通过 `conversationId` 绑定独立会话，多个 ChatPanel 可并行存在且互不干扰

**ChatPanel.vue 组件**
- 紧凑型对话面板UI组件，暴露 `mode`（floating/embedded）和 `context` props
- 三态切换：悬浮按钮（最小化）→ 弹出面板（半屏）→ 全屏展开（跳转Chat.vue）
- 首页使用时传递当前浏览知识条目上下文，作业指引使用时传递步骤上下文
- 与 Chat.vue 全屏对话页共享 useChat，用户切换页面后历史会话不丢失

### 5.7 对话上下文管理

- **短期记忆**：同一对话最近 10 轮消息保持在 Prompt 中，超出部分做摘要压缩
- **长期记忆**：对话结束后，用户标记为"有用"的回复及对应问题自动存入"修正记录"，关联用户和对话 ID
- **跨对话上下文**：从首页或作业指引中调起的 ChatPanel，自动携带当前上下文（设备型号、当前步骤、浏览的知识条目等），但不与用户的其他对话混淆
- **上下文隔离**：每个对话会话独立，不同会话之间不共享上下文。同一用户的多个对话可并行存在

## 6. 异常与边界情况

| 场景 | 处理方式 |
| ---- | -------- |
| 用户注册时用户名已存在 | 提示用户名已被占用，请更换 |
| 用户登录时连续 5 次密码错误 | 账号锁定 30 分钟，提示联系管理员 |
| 用户上传图片格式不支持 | 提示支持的文件格式（JPG/PNG/WebP），单个文件不超过 10MB |
| 用户上传附件格式不支持 | 提示支持的格式（PDF/Word/Excel），单个文件不超过 50MB |
| 检索时输入为空或仅含特殊字符 | 提示请输入有效的故障描述 |
| 检索无结果 | 展示"未找到匹配结果"，建议更换关键词、尝试 AI 对话、或提交客服工单 |
| 大模型服务不可用（超时或宕机） | AI 对话功能降级提示"AI 服务暂时不可用，请稍后重试或提交客服工单"，知识检索仍可用（向量检索不依赖大模型） |
| 向量数据库不可用 | 检索降级为数据库关键词搜索，提示"语义检索暂不可用，当前为关键词匹配模式" |
| 设备型号下无知识条目 | 提示"该设备暂无专属知识库"，展示跨设备的通用知识作为候补 |
| 管理员编辑知识条目时，另一管理员同时编辑同一条目 | 后提交者收到冲突提示，展示对方修改内容，需手动合并或覆盖 |
| 管理员回滚知识版本时，条目已再次被编辑 | 提示版本已有更新，确认是否仍要回滚，回滚后生成新版本（不是覆盖当前版本） |
| 新注册用户长时间未审核（超过 3 天） | 仪表盘显示超时提醒 |
| 客服工单超过 24 小时未回复 | 仪表盘显示超时提醒，工单高亮标记 |
| 用户申请密码重置 | 用户联系管理员，管理员在用户管理中重置密码，用户使用临时密码登录后强制修改 |
| 数据库备份过程中系统重启 | 备份标记为"失败"，下次自动备份周期正常执行，管理员可手动补备份 |
| 文件上传过程中网络中断 | 前端检测上传失败，提示用户重新上传，已上传的不完整文件由定时任务清理 |
| 浏览器不支持 WebP 格式 | 后端提供格式转换，统一转为 PNG 输出 |
| 并发检索量超过 20 人 | 请求排队处理，返回"当前检索繁忙，预计等待 X 秒"，不影响系统稳定性 |

## 7. 验收标准

### 7.1 用户端验收标准

1. 新用户完成注册，管理员审核通过后，用户可使用账号密码登录
2. 用户在首页浏览知识库分类树，点击分类可过滤知识条目列表；输入故障描述文本搜索，返回相关检修知识列表，按相关度排序
3. 用户上传一张设备故障图片并输入补充描述，点击搜索，返回图文联合检索结果
4. 用户选择设备型号后搜索，返回结果仅包含该设备型号关联的知识
5. 用户点击知识条目进入详情页，查看完整内容，下载附件
6. 用户在 AI 对话页输入问题，AI 返回结构化回答（分析 → 原因 → 方案 → 来源）
7. 用户对 AI 回答进行反馈（有用/部分有用/无用），选择无用时可填写修正内容
8. 用户创建作业任务时选择设备型号和检修等级，系统自动匹配推荐流程模板，进入作业指引页后按步骤执行，合规校验项勾选确认，每步留痕；步骤中可打开内嵌AI面板携带步骤上下文提问
9. 用户完成案例上传（文本+图片+附件），提交后可在个人工作台查看审核状态
10. 用户在 AI 对话中连续两次反馈无用，系统提示提交客服工单，用户确认后提交
11. 用户提交客服工单后，管理员回复，用户收到回复并确认解决
12. 用户在个人工作台查看我的任务、我的上传、我的客服工单、我的修正记录
13. 用户在首页浏览知识库时点击右下角AI悬浮按钮，弹出ChatPanel紧凑对话面板，面板自动携带当前浏览的知识条目上下文，AI回复中引用知识条目超链接
14. 用户在作业指引执行步骤中点击内嵌AI按钮，ChatPanel弹出并携带当前步骤的设备型号、步骤内容和合规要求上下文

### 7.2 管理端验收标准

1. 管理员使用专用账号登录管理后台，查看仪表盘统计数据
2. 管理员在知识库管理页新增知识条目，填写内容并发布，在用户端可检索到
3. 管理员编辑已有知识条目，版本号递增，在版本历史中可查看旧版本
4. 管理员对比两个版本的差异，将当前版本回滚到历史版本
5. 管理员在审核队列中查看待审核案例，完成初审（通过/驳回/通过含修改）
6. 标记为经验型的案例，专家完成复审
7. 审核通过的案例自动纳入检索索引，用户端可检索到
8. 管理员在客服管理页查看待处理工单，回复用户，用户端收到回复
9. 管理员将已解决的工单纳入知识库，生成知识条目草稿
10. 管理员在用户管理页审核新注册用户，禁用/启用用户，修改角色，重置密码
11. 管理员在设备型号与模板管理页新增设备型号和检修流程模板，用户端可选到新设备和模板；用户创建任务时自动匹配对应模板
12. 管理员在模型配置页切换大模型，用户端 AI 对话使用新模型
13. 管理员在操作日志页查看和导出操作日志
14. 管理员在系统设置页执行数据备份
15. 管理员上传PDF检修手册，系统解析文本、导出内嵌图片并调用视觉模型生成图片描述，分块向量化后进入知识库审核队列

### 7.3 非功能性验收标准

1. 系统在 LoongArch 架构 + 银河麒麟 V10 上完整部署并运行（后端、数据库、向量库）
2. 文本检索响应时间 < 2 秒，图文联合检索 < 5 秒
3. 20 个用户同时执行检索操作，系统稳定无崩溃
4. 用户登录后仅能访问用户端页面，管理员登录后仅能访问管理端页面，权限隔离无泄漏
5. 关键操作（作业步骤确认、知识条目修改、权限变更）在操作日志中可追溯
6. 任意浏览器窗口宽度缩至 768px 时，作业指引页布局自适应，功能不受影响

## 8. 可扩展性设计

### 8.1 模型层抽象

- LLM 调用通过统一的 ModelAdapter 接口，封装不同模型（DeepSeek / 千问 / 本地模型）的差异
- 新增模型仅需实现 Adapter 接口并注册到模型配置列表，无需修改业务代码
- 模型参数（Temperature、Top-K 等）通过管理端动态配置，即时生效

### 8.2 知识库可扩展

- 知识源抽象为 KnowledgeSource 接口，当前支持：手册导入、用户上传、客服工单转化
- 新增知识来源（如 ERP 工单系统对接、第三方知识平台）仅需实现 KnowledgeSource 接口
- 设备型号、故障分类均为配置驱动，管理员可在管理端动态新增，无需改代码
- 向量数据库抽象为 VectorStore 接口，当前适配 ChromaDB，后续可切换为 Milvus 等

### 8.3 检修流程模板引擎

- 检修流程模板为结构化 JSON 配置，管理员通过步骤编辑器可视化配置
- 支持模板间的继承与变体（如"通用电机检修流程"派生出"XX 型号电机检修流程"）
- 合规校验项为可配置规则，管理员可在模板编辑器中动态添加/修改

### 8.4 API 版本化

- 后端 API 使用 `/api/v1/` 前缀
- 当接口发生不兼容变更时，新增 `/api/v2/` 路由，保留旧版本至少一个大版本周期
- 前端屏蔽 API 版本差异，通过统一的 HTTP 客户端封装

### 8.5 前端组件化与 Composable 模式

- 通用组件（搜索栏、知识卡片、步骤导航、反馈组件）封装为独立 Vue 组件
- AI 对话能力封装为 `useChat` composable + `ChatPanel.vue` 组件，支持在多个页面中以不同形态嵌入，新增AI对话场景无需重写对话逻辑
- 页面路由使用懒加载，新增页面仅需添加路由配置
- UI 组件库通过统一封装层引入，后续更换 UI 库时仅需修改封装层

### 8.6 部署架构预留

- 后端设计为无状态服务，支持水平扩展（Nginx 负载均衡 + 多实例）
- 文件存储预留对象存储接口，当前使用本地文件系统，后续可切换为 MinIO
- 系统预留"离线知识包"导出功能接口，供未来移动端离线使用

## 10. 技术约束与假设

- **前端**：Vue 3 + Vite，Composition API + `<script setup>` 语法；AI对话能力通过 `useChat` composable + `ChatPanel.vue` 组件实现复用
- **后端**：FastAPI (Python 3.10+)
- **大模型编排**：LangChain，用于模型调用与检索链路编排
- **PDF解析**：PyMuPDF (fitz) — 文本提取、表格提取、内嵌图片导出；若LoongArch不兼容则备选 pdfplumber
- **向量数据库**：ChromaDB（需验证 LoongArch 兼容性，备选自编译 Milvus Lite）
- **关系数据库**：开发阶段使用 SQLite，生产环境 PostgreSQL（需验证 LoongArch 可用性）
- **文件存储**：本地文件系统（预留 MinIO 接口）；图片/附件先上传落盘再通过URL引用，数据库不存文件本体
- **检索架构**：知识图谱 + 向量检索双引擎
- **API 设计**：RESTful，`/api/v1/` 前缀，版本化管理
- **大模型API**：开发阶段通过 `.env` 环境变量配置API Key；最终发布时支持管理员在后台动态配置（云端API Key或本地部署Endpoint），配置变更即时生效无需重启
